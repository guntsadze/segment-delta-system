🧠 EvaluatorService:

1. არქიტექტურული პრინციპი: "Database First"
   ამ კოდის მთავარი ფილოსოფია არის ის, რომ მონაცემების ფილტრაცია ხდება ბაზაში და არა Node.js-ში.
   რატომ? თუ გვყავს 1,000,000 მომხმარებელი, მათი წამოღება Node.js-ში და filter()-ის გაკეთება გაავსებს მეხსიერებას (RAM) და "დაკრაშავს" სერვერს.
   გადაწყვეტა: ჩვენ ვიყენებთ ბაზის (PostgreSQL) სიმძლავრეს. ბაზა არის ოპტიმიზებული მილიონობით ჩანაწერზე აგრეგაციების (SUM, COUNT) გასაკეთებლად.

2. მეთოდი evaluate(segmentId)

ეს არის მთავარი "შესასვლელი" წერტილი.

სეგმენტის ამოღება: ჯერ ვამოწმებთ, საერთოდ არსებობს თუ არა ასეთი სეგმენტი.
სტატიკური (STATIC) სეგმენტის ლოგიკა:

if (segment.type === 'STATIC') {
const members = await this.prisma.segmentMembership.findMany({ ... });
return new Set(members.map((m) => m.customerId));
}

რატომ? თუ ის სტატიკურია, ჩვენ აღარ ვატარებთ მას ტრანზაქციების ფილტრში.
ჩვენ უბრალოდ ვამბობთ: "ვინც უკვე არის ამ სიაში, ის დააბრუნე". ეს უზრუნველყოფს მონაცემთა სიმყარეს (Data Integrity).

3. მეთოდი resolveCondition(condition)
   ეს არის "თარჯიმანი". ის იღებს ერთ კონკრეტულ პირობას და აბრუნებს იმ მომხმარებლების ID-ებს, რომლებიც მას აკმაყოფილებენ.
   ა) MIN_TRANSACTIONS_IN_DAYS & MIN_SPEND_IN_DAYS
   აქ ვიყენებთ Raw SQL-ს ($queryRawUnsafe).
   რატომ Raw SQL და არა Prisma-ს ჩვეულებრივი ფუნქციები?
   Prisma-ს სტანდარტულ ფუნქციებს უჭირთ HAVING და GROUP BY ოპერაციების შესრულება რთულ WHERE პირობებთან ერთად. SQL გვაძლევს სრულ კონტროლს.
   ლოგიკა:

WHERE "createdAt" >= NOW() - INTERVAL '${condition.days} days': ვფილტრავთ ტრანზაქციებს დროის მიხედვით.
GROUP BY "customerId": ყველა ტრანზაქციას ვაჯგუფებთ მომხმარებლის მიხედვით.
HAVING COUNT(\*) >= ...: ვტოვებთ მხოლოდ იმ მომხმარებლებს, რომლებსაც აქვთ საკმარისი რაოდენობის ტრანზაქცია.
HAVING SUM("amount") >= ...: იგივე ლოგიკაა დანახარჯზე.

ბ) INACTIVE_AFTER_ACTIVE
ეს არის ბიზნეს-ლოგიკა "მიძინებული" კლიენტების საპოვნელად.
SQL: SELECT "id" FROM "Customer" WHERE "lastTransactionAt" < NOW() - INTERVAL ...
ჩვენ ვეძებთ მომხმარებლებს, რომელთა ბოლო აქტივობა იყო უფრო ადრე, ვიდრე X დღის წინ.

გ) IN_SEGMENT (Cascade / Recursion-like logic)
ეს არის ყველაზე "ჭკვიანი" ნაწილი.
ეს საშუალებას გვაძლევს ავაწყოთ სეგმენტები ერთმანეთზე (როგორც Lego-ს კუბიკები).
თუ სეგმენტი "B" დამოკიდებულია სეგმენტ "A"-ზე, ჩვენ უბრალოდ ამოვიღებთ სეგმენტ "A"-ს ამჟამინდელ წევრებს.

4. მეთოდი applyOperator(operator, results)
   პირობების შეჯამება (Set Theory).
   წარმოიდგინე, გვაქვს ორი პირობა:
   ვინც იყიდა ფეხსაცმელი (Set A: {გიორგი, ნიკა})
   ვინც იყიდა მაისური (Set B: {ნიკა, ანა})
   AND (Intersection): results.reduce((a, b) => new Set([...a].filter(x => b.has(x))))
   პოულობს თანაკვეთას. შედეგი იქნება მხოლოდ {ნიკა}.
   OR (Union): results.reduce((a, b) => new Set([...a, ...b]))
   აერთიანებს ყველას. შედეგი იქნება {გიორგი, ნიკა, ანა}.
   რატომ Set და არა Array?
   Set ავტომატურად შლის დუბლიკატებს.
   Set.has() ოპერაცია არის O(1) სისწრაფის, მაშინ როცა მასივში includes() არის O(n). დიდ მონაცემებზე Set ბევრად სწრაფია.
